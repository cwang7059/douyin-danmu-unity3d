# 单位运动与动画规范

更新时间：2026-05-27

本文记录本项目在“单位看起来正常运动”上的开发结论，供后续替换模型、接入动画和优化战斗表现时参考。

## 1. 当前问题结论

当前 `Assets/StreamingAssets/PolyPizza/soldier.glb` 是静态模型：

- `animations = 0`
- `skins = 0`

这表示模型没有动画片段，也没有骨骼绑定。代码只能移动整个 GameObject，最多做轻微上下浮动、身体摇摆或影子变化，但无法让腿真正迈步。所以当前士兵视觉上会像“平移滑行”，不是正常走路。

真正解决方式不是继续硬写位移特效，而是换成带骨骼和行走/跑步动画的士兵资源。

## 2. 推荐实现方案

### 正式方案：Animator + 带骨骼模型

士兵、兽人、人形英雄等双足单位应使用 Unity 标准动画流程：

1. 导入带骨骼的角色模型，优先选择 `Humanoid` Rig。
2. 导入或绑定动作片段：`Idle`、`Walk`、`Run`、`Attack`、`Hit`、`Death`。
3. 为单位建立 `Animator Controller`。
4. 代码只控制单位逻辑位置，Animator 控制身体动作。
5. 根据移动速度驱动动画参数。

推荐 Animator 参数：

```text
Speed        float   0=站立，0.3=走路，1=跑步
Attack       trigger 攻击动作
Hit          trigger 受击动作
Dead         bool    死亡状态
MoveBlend    float   可选，区分走/跑混合
```

代码驱动示例：

```csharp
float normalizedSpeed = Mathf.Clamp01(unit.moveSpeed / unit.speed);
animator.SetFloat("Speed", normalizedSpeed, 0.12f, Time.deltaTime);

if (unit.attackVisualTimer > 0f)
{
    animator.SetTrigger("Attack");
}

animator.SetBool("Dead", !unit.active || unit.hp <= 0f);
```

### 不推荐：只靠代码摆动静态模型

静态模型可以临时加以下表现，但只能缓解，不能真正解决：

- 身体上下轻微 bob。
- 左右倾斜模拟重心。
- 脚底影子交替缩放。
- 枪口、武器、背包等局部摆动。

这些方式适合远景小兵或临时原型，不适合近景可见角色。

## 3. 动画资源标准

导入士兵或人形单位前必须检查：

| 检查项 | 要求 |
| --- | --- |
| 骨骼 | 必须有 Skin/Armature/Bone |
| 动画 | 至少包含 Walk 或 Run |
| 朝向 | 模型前方向要统一，必要时只在 `ModelRoot` 上修 yaw |
| 比例 | 士兵高度应明显小于坦克和怪兽 |
| 许可证 | 保存来源、作者、许可证到 `doc/许可证/` |
| 性能 | 大量单位要准备 LOD 或低频动画方案 |

GLB 快速检查思路：

- `animations` 数量大于 0，说明有动画片段。
- `skins` 数量大于 0，说明有骨骼蒙皮。
- 如果两个都是 0，基本就是静态模型，不适合作为近景行走士兵。

## 4. 单位类型运动规范

### 士兵

- 近景士兵必须使用骨骼动画。
- 移动时播放 `Walk` 或 `Run`。
- 攻击时上半身或全身播放 `Attack/Shoot`。
- 实际移动速度要和动画速度匹配，避免脚底打滑。
- 大量士兵可按距离降级：近处完整动画，远处低频更新，超远处简化模型。

### 坦克

- 坦克整体由代码控制位置和朝向。
- 坦克必须朝移动方向行驶，本项目人族坦克默认向右推进。
- 炮塔和炮筒应优先使用模型自己的部件，不要再用外部假炮筒覆盖真实模型。
- 履带和轮子应有运动感：
  - 模型有轮子节点时旋转轮子节点。
  - 模型有履带材质时滚动贴图 offset。
  - 没有明确节点时，可以增加很薄的辅助履带/轮子表现。

### 飞机/直升机

- 飞机整体可由代码控制巡航、摇摆和高度。
- 直升机螺旋桨必须持续旋转。
- 如果模型没有独立螺旋桨节点，可以挂一个透明旋翼辅助层。

### 怪兽/四足单位

- 四足怪物不要强行套 Humanoid 动画。
- 优先使用模型自带动画或专门的四足动画。
- 如果模型无动画，只能做临时程序化摇摆，但近景会显得滑。
- 大体型怪兽要有更大的碰撞/分离半径，避免多只怪物完全重叠。

## 5. Root Motion 使用建议

本项目是弹幕式大规模自动战斗，建议默认不启用 Root Motion：

- 单位位置由战斗逻辑、寻敌、碰撞分离和阵型控制。
- Animator 只负责动作表现。
- 动画速度通过参数和播放速度同步到逻辑速度。

只有在英雄演出、冲锋技能、击退动作这类短时间强表现动作中，才考虑局部启用 Root Motion。

## 6. 后续开发清单

1. 替换 `soldier.glb` 为带骨骼和 Walk/Run 动画的士兵模型。
2. 在 Unity 中设置 Rig，并确认动画能正常播放。
3. 新增 `Animator Controller`，绑定 `Idle/Walk/Run/Attack/Death`。
4. 在 `ApocalypseKingUnityGame` 的单位挂载逻辑里缓存 `Animator`。
5. 用 `unit.moveSpeed` 驱动 `Speed` 参数。
6. 攻击和死亡时触发对应动画。
7. 运行后观察运动方向，确认脚步方向和单位移动方向一致。
8. 截图或日志验证后立即删除临时文件，不提交验证产物。

## 7. 验证标准

每次替换单位模型或动画后都必须检查：

- 士兵移动时是否迈腿，而不是整体滑行。
- 士兵朝向是否和前进方向一致。
- 攻击时是否进入攻击动作。
- 坦克是否向右推进，炮筒是否来自模型自身。
- 直升机螺旋桨是否旋转。
- 怪兽是否有走路/爬行动作或至少有合理步态表现。
- 临时截图和日志是否已清理。
